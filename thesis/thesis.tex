Congested Clique, described in \cite{powerCongestClique} is a mathematical model for distributed computing. It features $n$ inter-connected players in a clique network, sending messages in a synchronized manner. In each round, each player sends $O(\log{n})$ to each other player. The computation efficiency is measured by the communication rounds. 
In this work, the congested clique algorithm that solves the $MST$ in $O(\log{\log{n}})$ rounds \cite{MSTLogLogn} is studied and implemented on the supercomputer Puhti \cite{Puhti}. We simulated the congested clique model with CPU threads and a CUDA GPU to run the $MST$ algorithm. 
The purpose of this work is to identify the connection between $A$ and modern distributed computing hardware and tools. The relations allow us to understand the bigger picture as $A$ is representative of the congested clique model

The representativeness of $A$ can be shown in the following ways:
\begin{itemize}
	\item[--] $A$ exploit the full connectivity of the graph. In each round, there is a step where each player sends $O(\log{n})$ messages to all the other players.
	\item[--] To comply with the $O(\log{n})$ limit when sending to a unique leader, large data is spilled to other players before sending.
	\item[--] The algorithm needs several rounds before its convergence.
\end{itemize}

We measure how compatible $A$ is with the Supercomputing infrastructure to find the connections. The compatibility is measured when writing the code to simulate the congested clique with the hardware. 

\begin{itemize}
	\item[--] How to simulate the local computation.
	\item[--] How to simulate the message passing.
\end{itemize}
Secondly, how the hardware is limiting the performance of the algorithm. Memory, CPU, GPU, network, which one is the bottleneck when the algorithm scales.
How the algorithm performs compared to traditional sequential algorithms. By increasing the number of nodes, will it outperform the sequential algorithm with
\begin{itemize}
	\item[--] Speed comparing to sequential algorithms.
	\item[--] Number of edges can be processed compared to sequential algorithms.
\end{itemize}
Scalability:
\begin{itemize}
	\item[--] Strong scaling and weak scaling (How having more processes speeds up the computation, which part of the hardware will become bottleneck when scaling continuously).
	\item[--] Time composition for the run when scaling (MPI\_time vs local). Does the time complexity hold?
\end{itemize}

\section{CPU implementation}
\subsection{Congested Clique simulation}
$A$ is simulated with OpenMPI and C++. Multiple CPU threads run cooperatively to simulate the congested clique model. Each thread has its dedicated memory space and computing power. Each thread simulates multiple machines in the congested clique model. The local computation of the congested clique is simulated by each thread's local computation while the communication is handled by OpenMPI.

MPI (Message-passing interface) is a message-passing library interface specification that message padding model for parallel computing \cite{MPI}. Open MPI is an open-source Message-passing Interface implementation. 

One CPU thread can hold one or multiple vertices. The thread number is limited. With each thread simulating more than one vertex, larger graphs can be simulated. To simulate more than one vertex, each thread holds all the edges adjacent to the vertices it is responsible for and run the algorithm for each vertex. 

When running $A$, at the beginning, each rank receives the adjacent edges of the node(s) it is responsible for. The algorithm is run iteratively until the MST is found by repeating steps. Each iteration consists of a fixed number of steps and MPI communication.

step 1 $\rightarrow$ MPI\_Alltoall\\
step 2 $\rightarrow$ MPI\_Alltoall\\
step 3 $\rightarrow$ MPI\_Scatter\\
step 4 $\rightarrow$ MPI\_Gather\\
step 5 and 6 $\rightarrow$ MPI\_Scatter and local computation\\

To ensure the correctness of the algorithm, the result of $A$ is compared to the result of the prim's algorithm on the same graph.

\subsection{Memory limit and graph generation}
Each node can have at most 40 threads and has a memory limit of 192 GiB. With $n$ threads for a node, each thread can have at most $192/n$ GiB memory.

The graph is a complete graph represented by a square adj matrix. Each edge is assigned a random weight. There is a naive approach and a distributed approach to generate the graph.

Naive generation: generate the adj matrix on rank 0 and scatter it to all other ranks with MPI\_Scatter. Each rank gets a matrix row. Memory eager ($O(n^2)$ memory needed), can't run large enough graph.

Distributed generation: Each rank generates a number of rows of the adj matrix. And share the result with the destination peer with MPI\_Alltoall. It saves the memory for graph storage and allows running larger graphs.

\subsection{performance compared to sequential algorithm}
Prim's algorithm is used as the sequential algorithm to compare the performance of $A$. It is implemented also in C++ and runs on a single thread on Puhti, with 16G memory.

With 16G memory, prim's algorithm can process 32678 vertices (but not 65356) and $32678 \times 32677 / 2 (5.33 \times 10 ^ 8)$ edges using 88 seconds. The bottleneck for prim's algorithm to process more edges is to fit the graph to the memory.
